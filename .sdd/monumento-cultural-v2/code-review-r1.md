## Medium priority issues (consider)

### M1. `api/src/api.ts:165-170` — `estimateCost` duplicated
```ts
function estimateCost(text: string, model: string): number {
  const outputTokens = Math.ceil(text.length / 4);
  const costPer1M = model.includes("reasoner") ? 2.19 : 0.14;
  return (outputTokens / 1_000_000) * costPer1M;
}
```
This is byte-for-byte identical to `api/src/llm/cost-tracker.ts:13-17`. The frozen `/v1/simulate` handler kept its local copy instead of importing the canonical one. Not a bug, just duplication that will rot (e.g., if pricing changes, only one is updated).

### M2. `api/src/api.ts:151-163` — `buildSystemPrompt` is a stub
The frozen `/v1/simulate` handler builds its own prompt with only 3 layers (identity, Big Five, style guard). The richer `buildContext()` in `api/src/context/builder.ts` (frozen, PR #9) has all 7 layers (identity + psychometric + RAG + style guard + injection defense). The stub should be replaced by a `buildContext()` call. The PR #9 dialogue handler already does this correctly.

### M3. `api/src/handlers/dialogue.ts:147` — hard-coded model in cost calc
```ts
cost: estimateCost(fullText, "deepseek-chat"),
```
The model string is hard-coded; if `ocaisStream` is invoked with a non-default model (`opts.model`), the cost estimate is wrong. The model is not propagated through the `OcaisChunk.done` envelope. Either:
- Add `model` to the `done` chunk in `provider.ts` (frozen).
- Pass the model through `opts` to the handler and use it here.

### M4. `api/src/rag/retrieve.ts:122-125` — per-call Float32Array conversion
```ts
const scored = corpus.map((doc) => ({
  doc,
  score: cosine(queryEmbedding, new Float32Array(doc.embedding)),
}));
```
Every `retrieve()` call converts every `doc.embedding` (`number[]`) to a `Float32Array`. The corpus artifact is loaded once per cold start (cached in `dialogue.ts:66`), so a one-time `Float32Array` conversion at corpus load would eliminate the per-request allocation. Minor perf win on the hot path; the R3 perf baseline (`tests/perf/benchmark.test.ts`) already passes <50ms so this is not urgent.

### M5. `api/src/state/persona-state.ts:65-73` — branching is over-clever
```ts
if (Array.isArray(partial.recentEvents)) {
  const evs = partial.recentEvents;
  merged.recentEvents =
    evs.length > RECENT_EVENTS_MAX
      ? evs.slice(evs.length - RECENT_EVENTS_MAX)
      : evs;
} else if (!merged.recentEvents) {
  merged.recentEvents = [];
}
```
The `Array.isArray(partial.recentEvents)` check returns true even for `[]`, so the `else if` is unreachable when `partial.recentEvents = []`. Behavior is correct but the logic reads as if there were a third branch that doesn't exist. Could be simplified to a single `if (Array.isArray(partial.recentEvents))` block.

### M6. `api/src/llm/rate-limiter.ts:65` — `size()` exposed but unused
```ts
size(): number {
  return this.buckets.size;
}
```
Only used in `tests/llm/rate-limiter.test.ts`. Production never calls it. If H2 above is acted on (wire `TokenBucket` into middleware), this becomes useful.

## Low priority issues (cosmetic)

### L1. `api/src/api.ts:129,147` — TODO without ticket
```ts
tokens_in: 0, // TODO: track desde ocais
// TODO S2: portar prompt_builder.py completo a TypeScript
```
"TODO S2" is not a ticket reference. Polish R7 created the `alarms.config.ts` deferred-wiring convention with the pattern `// deferred to R8 or whenever`. Recommend adopting that pattern here (e.g., `// TODO(opita-XXX):` with a real ticket key).

### L2. `api/src/api.ts:78-84` — request body parsed without validation
```ts
app.post("/v1/simulate", async (c) => {
  const body = await c.req.json<{
    city_id: string;
    persona_id: string;
    scene: { time: string; place: string; weather?: string };
    ...
  }>();
  if (body.city_id !== "tello") { ... }
  const persona = TELLO_PERSONAS.find((p) => p.persona_id === body.persona_id);
```
The `/v1/simulate` handler reads `body.city_id`, `body.persona_id`, `body.scene.time`, etc. directly. If the client sends `{}` or a non-object, `body.city_id` throws (cannot read property of undefined). The new `validateDialogueRequest()` is used for `/v1/dialogue` but not here. Should reuse the same validation pattern or define a parallel one.

### L3. `api/src/handlers/dialogue.ts:187` — `void personaState;` smell
```ts
void personaState;
```
Comment says "marked it referenced so tsc does not complain while we leave the hook in place for Phase 2." Cleaner options:
- Prefix with `_`: `const _personaState = ...` and never reference it.
- Add a `// Phase 2: feed personaState into context` comment with a real ticket.
- Remove the load entirely until Phase 2 lands.

### L4. `api/src/state/ventana-events.ts:78-80` — string comparison of ISO timestamps
```ts
if (options.since && ts < options.since) return false;
if (options.until && ts > options.until) return false;
```
Works correctly only when both `ts` and the option values are Z-normalized ISO strings. If a caller passes a non-normalized string (e.g., missing the trailing `Z`), the comparison is wrong but silent. Defensive `new Date(...).toISOString()` coercion in the public API would close this.

### L5. `web/scripts/analyze-bundle.ts:245-322` — emoji-heavy output
The script uses 📦, ⚠️, ✅ in console output. Functional but reduces portability (Windows legacy terminals without UTF-8). Cosmetic; the operator explicitly chose this style in R3.

### L6. `api/src/rag/embed-query.ts:42` — fallback cache dir uses `USERPROFILE ?? HOME ?? "."`
The `"./"` fallback can cause model files to be cached in the Lambda container's working directory, which is ephemeral. SST bundles the function to `/tmp` by default. Defensive: assert that `cacheDir` is writable, or fail-fast with a clear error pointing to the operator.
